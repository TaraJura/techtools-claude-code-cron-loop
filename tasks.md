Calculate the scale factor as `min(newWidth/oldWidth, newHeight/oldHeight)`. (2) **Stretch to fill** — non-uniformly scale content to exactly fill the new page dimensions (distorts content if aspect ratios differ — warn the user). (3) **Center, no scale** — place the original content centered on the new page without any scaling. If the new page is larger, content appears with extra margins. For annotations and form fields, adjust their positions proportionally based on the scale factor and offset. **Page scope**: Let users apply the resize to: (1) the current page only, (2) all pages, (3) a custom page range (e.g., "1-5, 8, 12-20"), or (4) only pages matching a specific current size (e.g., "resize all Letter pages to A4" — auto-detect by comparing page dimensions to known standard sizes within a 2pt tolerance). Default margins: 0 (no extra margins beyond what scaling produces). **Live preview**: Show a real-time preview of the resized page in the panel, rendered at a reduced resolution. Display the original page outline (dashed border) overlaid on the new page dimensions (solid border) to visualize how content maps to the new size. Update the preview on every parameter change with a 100ms debounce. Include a before/after toggle or split view. Show the scale percentage applied (e.g., "Content scaled to 94.3%"). **Batch consistency**: When resizing all pages, handle documents with mixed page sizes gracefully — each page's scale factor is calculated independently based on its original dimensions. Show a warning if the document contains pages of different sizes: "This document has 3 different page sizes. Each will be scaled independently to the target size." Offer an option to "Make all pages uniform" which is a common use case (e.g., a scanned document with slightly varying page sizes). **Bleed and trim**: For print-production users, offer optional bleed settings — extend the page dimensions by a specified bleed amount (default 3mm) on each edge, commonly needed for professional printing. Display trim marks at the original page boundary corners as thin lines extending into the bleed area, drawn using pdf-lib's `page.drawLine()`. This is valuable for users preparing files for commercial print shops. **UI integration**: Add a "Resize Pages" button in the Pages tab/toolbar. The resize panel shows: target size selector, orientation toggle, content handling mode selector, page scope selector, margin inputs (collapsible "Advanced" section), bleed settings (collapsible "Print Production" section), and the live preview. A "Resize" button applies the changes, producing `[filename]-resized.pdf`. Keep the original in memory for undo. **Keyboard accessibility**: Tab through all controls, ARIA labels, Enter to apply. **Edge cases**: Handle pages with zero-width or zero-height (skip with warning), extremely large pages (cap at 14400pt), pages with crop/media/bleed box mismatches (use the media box as the authoritative size), rotated pages (account for `/Rotate` attribute — a 90° rotated page has its effective width and height swapped), encrypted PDFs (prompt for password), and PDFs with page-level form fields or link annotations (scale annotation rectangles proportionally). ### TASK-072: PDF font inspector and embedding tool — view, analyze, and fix font issues

**Status**: DONE
**Priority**: MEDIUM
**Assigned to**: developer2
**Description**: Build a font inspection and management tool that lets users view all fonts used in a PDF document, check their embedding status, identify potential rendering issues, and fix missing or non-embedded fonts — one of the most common causes of PDFs displaying incorrectly across devices. Font problems are the #1 support issue in professional PDF workflows: a document looks perfect on the author's machine but renders with wrong characters, missing glyphs, or substituted fonts on the recipient's device. **Font extraction and analysis**: Use pdf.js to parse each page's font resources. For every page, call `page.getOperatorList()` and `page.commonObjs` / `page.objs` to collect font references. For each font, extract: (1) **Font name** — the `/BaseFont` value (e.g., "TimesNewRomanPSMT", "ArialMT", "BCDFGH+Helvetica-Bold"). Detect subset prefixes (6 uppercase letters + "+" prefix like "BCDFGH+") which indicate the font is a subset embedding. (2) **Font type** — Type1, TrueType, Type3, CIDFontType0 (CFF/OpenType), CIDFontType2 (TrueType-based CID), or Type0 (composite). Display in human-readable form (e.g., "TrueType", "OpenType/CFF", "Type 1 PostScript"). (3) **Encoding** — WinAnsiEncoding, MacRomanEncoding, Identity-H, Identity-V, or custom. Flag non-standard encodings that may cause character mapping issues. (4) **Embedding status** — fully embedded (complete font program present), subset embedded (only used glyphs included — detected by the subset prefix), or not embedded (only font name referenced, relying on system fonts). This is the most critical piece of information. (5) **Glyph count** — for embedded fonts, count the number of glyphs in the font program. For subset fonts, compare against the number of unique characters actually used from that font in the document. (6) **Font flags** — parse the `/FontDescriptor` dictionary for: italic, bold, serif/sans-serif, fixed-pitch, symbolic, script, all-cap, small-cap. (7) **Unicode mapping** — check if the font has a `/ToUnicode` CMap. Fonts without ToUnicode may not support text extraction or copy-paste correctly — flag this as a warning. (8) **Pages used** — track which pages reference each font, so users can locate where specific fonts appear. **Font panel UI**: Add a "Fonts" tab in the sidebar or a "Font Inspector" tool in the toolbar. Display a sortable table listing all fonts in the document with columns: Font Name, Type, Embedding Status (with color-coded badges — green for "Embedded", yellow for "Subset", red for "Not Embedded"), Pages Used, and a details expand arrow. Show summary statistics at the top: total font count, embedded count, subset count, not-embedded count, and an overall "Font Health" indicator (green if all fonts are embedded, yellow if subsets only, red if any fonts are not embedded). Add a search/filter box to quickly find fonts by name. Group fonts by type or embedding status using tabs or a dropdown filter. **Font details view**: Clicking a font row expands a detailed panel showing: all extracted properties (type, encoding, flags, glyph count), a character map preview (render the first 100–200 available glyphs in a grid using a small canvas — use pdf.js's font rendering to display actual glyphs as they appear in the PDF), the list of pages where this font is used (clickable to navigate to that page), and a sample text preview showing how text in this font actually renders. For subset-embedded fonts, show which specific characters are included in the subset. **Problem detection**: Automatically scan for and flag common font issues: (1) **Not embedded fonts** — the biggest problem; these fonts will be substituted by the viewer, often incorrectly. Show a red warning with the explanation "This font is not embedded — it will be replaced by a system font on devices that don't have it installed, potentially changing the document's appearance." (2) **Missing ToUnicode map** — text using this font cannot be reliably searched or copied. Show a yellow warning. (3) **Type 3 fonts** — these are custom bitmap fonts that often cause rendering issues across viewers. Flag with an informational notice. (4) **Symbolic fonts with non-standard encoding** — may cause character substitution issues. (5) **Subset fonts with very few glyphs** — if a subset contains fewer than 10 glyphs, note that this is a minimal subset (fine, but worth knowing). (6) **Duplicate font names** — multiple font objects with the same base name but different properties, which can indicate conflicts. Display all detected issues in a "Font Issues" summary section at the top of the panel, sorted by severity (errors first, then warnings, then info). **Font embedding fix**: For non-embedded fonts, offer a "Fix: Embed Font" action. Implementation approach: since true font embedding requires access to the actual font file (which may not be available in the browser), provide two strategies: (1) **Render and replace** — for each page using the non-embedded font, render the text content to a canvas at high resolution using pdf.js (which can often render the font if it matches a standard name), then overlay the rendered image. This preserves appearance but converts text to images on affected pages. Warn that this makes the text non-searchable on those pages. (2) **Substitute with web font** — map the non-embedded font name to the closest available open-source web font (e.g., map "Arial" → Liberation Sans, "Times New Roman" → Liberation Serif, "Courier New" → Liberation Mono, "Helvetica" → Nimbus Sans) from a bundled set of open-source metric-compatible fonts loaded via JavaScript. Use pdf-lib to embed the substitute font and remap the text. Show a before/after preview so users can verify the substitution looks acceptable. Include a font mapping configuration panel where users can choose which substitute font to use for each non-embedded font. **Export font report**: Generate a downloadable font report as PDF or JSON containing: document filename, analysis date, complete font inventory with all properties, detected issues with severity and descriptions, and recommendations. Useful for pre-flight checks before sending documents to print or archive. **PDF/A compliance check**: Since PDF/A (archival format) requires all fonts to be embedded, add a quick "PDF/A Font Compliance" check that reports whether the document meets font embedding requirements for PDF/A-1b, PDF/A-2b, and PDF/A-3b standards. Show a pass/fail indicator with specific non-compliant fonts listed. **Keyboard shortcut**: Register with the command palette (TASK-048) as "Inspect Fonts" and "Font Report". **Performance**: Scan fonts lazily — parse the first page immediately for quick feedback, then scan remaining pages in the background using `requestIdleCallback`. Show a progress indicator ("Scanning page 15/42...") during the background scan. Cache font analysis results so toggling back to the Fonts tab doesn't re-parse. For very large documents (>200 pages), offer a "Quick Scan" that samples every 5th page and a "Full Scan" for comprehensive analysis. **Edge cases**: Handle PDFs with no fonts (scanned image-only documents — show "No fonts detected, this may be a scanned document"), encrypted PDFs (warn that font analysis may be limited), PDFs with hundreds of fonts (paginate the font list, add virtual scrolling), Type 3 fonts with custom drawings instead of standard glyphs (render the custom glyphs in the preview), CJK fonts with very large glyph sets (show glyph count but limit the character map preview to a representative sample), and fonts with corrupted or incomplete font programs (flag as "Damaged font" with an error indicator). All processing client-side in the browser using pdf.js for font parsing and pdf-lib for font embedding modifications.
