For page operations (TASK-007, TASK-024, TASK-020): wrap page reorder, delete, insert, rotate, and crop in page commands. For text overlays (TASK-015): wrap text addition and editing. For watermarks (TASK-013): wrap watermark application. For form filling (TASK-009): wrap field value changes. For signatures (TASK-010): wrap signature placement. For freehand drawing (TASK-022): wrap stroke completion (not individual mouse move events — capture the complete path as one command). For stamps (TASK-028): wrap stamp placement. This integration requires each module to expose a way to programmatically add/remove/modify its artifacts — most modules already do this via their internal state management, so the command's `undo()` method calls the module's removal/restoration API. **Keyboard shortcuts**: Bind `Ctrl+Z` (Windows/Linux) / `Cmd+Z` (Mac) to undo and `Ctrl+Y` or `Ctrl+Shift+Z` / `Cmd+Shift+Z` to redo. Register these with the command palette (TASK-048) as "Undo" and "Redo" commands. Ensure shortcuts don't conflict with browser-native undo in text input fields — when a text input or textarea is focused, let the browser handle undo natively; only intercept when the PDF canvas or general UI is focused. **Toolbar buttons**: Add undo (counterclockwise arrow icon) and redo (clockwise arrow icon) buttons to the main toolbar, positioned prominently near the left side. Disable/gray out the buttons when their respective stacks are empty. Show a tooltip with the operation description on hover: "Undo: Added highlight on page 3" or "Redo: Deleted page 5". Add a small dropdown arrow next to the undo button that opens a mini-list of the last 10 undoable operations — clicking any entry undoes everything up to and including that operation (a multi-step undo). **History panel UI**: Add a "History" panel accessible from the left sidebar (alongside thumbnails, bookmarks, annotations panels). The panel shows a vertical timeline of all operations in chronological order (newest at top). Each entry displays: an icon representing the operation type (highlight icon for annotations, page icon for page ops, text icon for text overlays, etc.), the operation description ("Added sticky note on page 2", "Rotated page 5 by 90°", "Merged 3 pages from document.pdf"), the timestamp (relative — "2 minutes ago", "just now"), and affected page numbers as clickable badges (clicking navigates to that page). The current position in history is indicated by a horizontal divider line — entries above are the current state (executed), entries below (if any) are on the redo stack and shown grayed out/dimmed with a "future" indicator. Clicking any entry in the history jumps to that state — undoing or redoing multiple operations in sequence to reach that point, with a brief progress indicator if many operations need to be replayed. Include a "Clear History" button at the bottom (with confirmation dialog) and a counter showing "47/100 operations" (current/max). **Memory management**: Since storing full page content for page delete undo could consume significant memory (each PDF page can be hundreds of KB), implement a tiered storage approach: for lightweight operations (annotations, form fills, text overlays), store the full data in memory; for heavyweight operations (page deletions, large image insertions), store the page data in a `Blob` and keep only a reference — if memory pressure is detected (via `performance.memory` API where available, or by tracking total command payload size), convert older heavyweight undo entries to a compressed format or drop them with a warning in the history panel ("Some older operations can no longer be undone — memory limit reached"). Set a configurable memory budget (default 50MB for undo history). **Session persistence**: Optionally persist the undo history to IndexedDB (building on TASK-036's auto-save infrastructure) so that if the user refreshes the page or the tab crashes, the editing history survives. Show a "Restore previous session?" prompt that includes the operation count: "You have 23 unsaved editing operations from your previous session. Restore?" **Edge cases**: Handle concurrent modifications gracefully — if the user undoes an annotation addition but another annotation was added on top of it (overlapping), ensure z-order is preserved correctly.
