### TASK-073: Export PDF to Markdown and plain text with intelligent formatting preservation

**Status**: DONE
**Priority**: MEDIUM
**Assigned to**: developer2
**Description**: Build a PDF-to-Markdown and PDF-to-plain-text export feature that converts PDF content into clean, well-structured Markdown (.md) or plain text (.txt) files, intelligently preserving document formatting such as headings, lists, tables, bold/italic emphasis, and hyperlinks. This completes the conversion toolkit alongside PDF-to-HTML (TASK-054), PDF-to-DOCX (TASK-038), PDF-to-PPTX (TASK-063), PDF-to-Images (TASK-014), and Table-to-CSV (TASK-037) — and addresses a high-demand use case for developers, writers, and knowledge workers who need to extract PDF content for use in wikis, static site generators, note-taking apps (Obsidian, Notion), README files, and documentation systems. **Text extraction and structural analysis**: Use pdf.js `page.getTextContent()` to extract all text items with their position (x, y), dimensions (width, height), font size, font name, and transform matrix. Analyze the extracted items to reconstruct the document's logical structure: (1) **Heading detection** — identify text with significantly larger font sizes relative to the dominant body font size (compute the mode font size across the document as the "body" baseline). Map font sizes to Markdown heading levels: text >= 2× body size → `# H1`, >= 1.7× → `## H2`, >= 1.4× → `### H3`, >= 1.2× → `#### H4`. Also detect headings by bold font weight when they appear on their own line with vertical spacing above/below. (2) **Paragraph reconstruction** — group text items into paragraphs by analyzing vertical gaps. Items on consecutive lines with Y-gaps < 1.5× the line height belong to the same paragraph. Merge hyphenated words at line breaks (detect trailing hyphens and join with the first word of the next line). Normalize whitespace within reconstructed paragraphs. (3) **List detection** — identify lines starting with bullet characters (•, -, ●, ○, ▪, ▸, ►) → `- ` unordered list items. Detect numbered patterns (1., 2., a., b., i., ii., (1), (a)) → `1. ` ordered list items. Detect nested lists by indentation level — items indented further than the parent list level become sub-items with additional indentation. (4) **Table detection** — reuse column alignment analysis from TASK-037's table extraction logic. Detect grid-aligned text clusters and convert to Markdown table syntax with `|` column separators and `---` header separators. Right-align numeric columns, left-align text columns. Handle merged cells by spanning with empty cells. (5) **Bold and italic detection** — parse font names for weight/style indicators: names containing "Bold", "Bd", "Heavy", "Black" → wrap text in `**bold**`. Names containing "Italic", "It", "Oblique" → wrap in `*italic*`. Names containing both → `***bold italic***`. (6) **Link detection** — extract link annotations from each page's `/Annots` array, match their bounding rectangles to the text items they overlay, and convert to Markdown `[link text](URL)` syntax. For internal links (page references), convert to anchor references. (7) **Code block detection** — identify text in monospaced fonts (Courier, Consolas, "Source Code Pro", or fonts with the FixedPitch flag set) that forms contiguous blocks. Convert to fenced code blocks (```) in Markdown. Single inline monospaced spans become inline code (`code`). (8) **Image placeholders** — detect images on each page via the operator list (`paintImageXObject`). In Markdown mode, insert `![Image on page N](image-N-M.png)` placeholders. In the multi-file export, extract the actual images and include them alongside the Markdown file. In plain text mode, insert `[Image]` placeholders. **Output formats**: (1) **Markdown (.md)** — full GitHub-Flavored Markdown with headings, emphasis, lists, tables, links, code blocks, and horizontal rules (detected from thin full-width drawn lines). Include a YAML frontmatter block with metadata: `title` (from PDF metadata), `author`, `date`, `source` (original filename), and `pages` (page count). (2) **Plain text (.txt)** — clean text with structural hints: headings in ALL CAPS or underlined with `===`/`---`, lists preserved with `-` and number prefixes, tables formatted with fixed-width columns using spaces, emphasis indicated with _underscores_ or *asterisks* (configurable). No Markdown syntax — optimized for readability in any text editor. (3) **Markdown + images (.zip)** — a ZIP package containing the `.md` file plus an `images/` folder with all extracted images as PNG files, properly referenced from the Markdown. **Customization options**: Page range selection (all, current page, custom range like "1-5, 8, 12-15"). Toggle headings detection on/off. Toggle list detection on/off. Toggle table detection on/off (some tables are better as plain text). Toggle link extraction on/off. Image handling: "Include placeholders" (default), "Extract images" (for ZIP export), or "Omit images". Line wrapping for plain text: "None" (single long lines per paragraph), "Wrap at 80 characters", "Wrap at 120 characters", or custom width. Page separator: "Page break marker" (`---` or `<!-- page break -->`), "Blank lines", or "None" (continuous text). Header/footer stripping: auto-detect repeated text at the top/bottom of pages (same text appearing on >50% of pages in the same position) and offer to strip it — removes page numbers, running headers, and footers that clutter the extracted text. **Smart cleanup**: Apply post-processing to clean up common extraction artifacts: remove soft hyphens (U+00AD) and join hyphenated words across lines, normalize Unicode characters (e.g., replace fancy quotes with straight quotes, em-dashes with `---`, en-dashes with `--`), collapse excessive whitespace and blank lines (max 2 consecutive blank lines), fix common OCR artifacts if the text appears to come from a scanned document (isolated single characters, unusual spacing patterns). Detect and preserve intentional formatting like poetry or code where unusual spacing is meaningful (heuristic: if a block has irregular short lines that don't fill the text width, treat it as preformatted). **Live preview**: Show a real-time preview of the Markdown or plain text output in a panel alongside the PDF viewer. Render the Markdown preview using a simple Markdown-to-HTML converter for visual verification. Let users toggle between "Raw Markdown", "Rendered Preview", and "Plain Text" views. Update the preview when the user changes options (page range, toggles, etc.). Limit the live preview to the first 20 pages for performance — show a "Preview limited to 20 pages" notice for longer documents. **Copy to clipboard**: Add a "Copy to Clipboard" button alongside the download button — many users want to paste extracted text into another application rather than download a file. Copy the full output or just the visible preview selection. Show a toast notification "Copied to clipboard!" on success. **Progress and performance**: Show a progress bar during extraction (per-page progress). Process pages in batches of 10 using `setTimeout` to keep the UI responsive. Cache extracted text so re-exporting with different options doesn't re-parse the PDF. For very large documents (>100 pages), show estimated processing time and file size. **UI integration**: Add "Export to Markdown" and "Export to Text" buttons in the "Convert" tab panel alongside existing export options (HTML, DOCX, PPTX, images). The export panel shows format selection (Markdown/Plain Text), customization options in a collapsible "Options" section, the live preview area, and Download/Copy buttons. Name output files as `[original-filename].md` or `[original-filename].txt`. For the ZIP package: `[original-filename]-markdown.zip`. **Edge cases**: Handle scanned/image-only PDFs (detect via absence of extractable text items — show "No extractable text found. Run OCR first?" with a link to TASK-008's OCR tool), right-to-left text (detect from PDF's text direction metadata or character ranges — preserve RTL in Markdown using Unicode bidi markers), vertical text (CJK documents — extract and convert to horizontal reading order), multi-column layouts (detect columns by analyzing text item X-positions for bimodal distribution — extract left column first, then right column, separated by a page-section break), mixed content pages (interleaved text and images — maintain reading order and insert image placeholders at the correct positions), empty pages (skip or insert a blank-line separator), form fields (extract field values and present as "Field Name: Value" pairs), and PDFs with no text at all (image-only — suggest OCR). All processing client-side in the browser using pdf.js for text extraction — no server involvement.

For tiled image patterns, create a `/PatternType 1` (tiling pattern) with the image as the pattern cell content. Insert all background/border drawing operations at the beginning of the page's content stream (before existing content) so they render behind text and images. To do this with pdf-lib, use the technique of extracting the existing page content stream, prepending the new background drawing commands, and rewriting the combined stream to the page. Alternatively, use pdf-lib's built-in `page.drawRectangle()` / `page.drawImage()` with a lower z-order by manipulating the `Contents` array order. **Page scope**: Let users apply backgrounds and borders to: "Current page only", "All pages", "Odd pages only", "Even pages only", "Custom range" (e.g., "1-3, 5, 8-12"), or "Pages matching size" (apply only to pages with the same dimensions as the current page — useful for documents with mixed page sizes). Show a thumbnail strip of affected pages with a preview overlay. **Live preview**: Render a real-time preview of the background and border on the current page in the PDF viewer. Let users save their own custom presets to localStorage for reuse. **Remove existing backgrounds**: Detect and remove existing background elements from PDF pages. Below the tabs, show the page scope selector, preset buttons, and "Apply" / "Remove" / "Reset" buttons. Register with the command palette (TASK-048) as "Page Background", "Page Border", and "Add Border". **Keyboard shortcut**: No default shortcut (this is a visual editing feature best accessed via the panel). **Output**: Apply changes in-place to the loaded PDF. All processing client-side in the browser using pdf-lib for final embedding — no server involvement.

### TASK-077: PDF/A archival format conversion and validation

**Status**: DONE
**Priority**: MEDIUM
**Assigned to**: developer2
**Description**: Add a PDF/A compliance checker and converter that lets users validate whether a PDF conforms to PDF/A standards (PDF/A-1b, PDF/A-2b, PDF/A-3b) and convert regular PDFs to PDF/A format for long-term archival — a critical feature for legal, government, and enterprise users who need archival-grade documents. **Validation panel**: Add a "PDF/A Compliance" tool accessible from the main toolbar (archive/shield icon) and the command palette (TASK-048). When activated, the tool analyzes the currently loaded PDF and produces a compliance report. The checker should inspect: embedded fonts (all fonts must be embedded — flag any that are missing or subset-only when full embedding is required), color spaces (PDF/A-1b requires device-independent color spaces — flag RGB/CMYK usage without ICC profiles), transparency (PDF/A-1b forbids transparency — flag any transparent objects), JavaScript and actions (PDF/A forbids JavaScript, launch actions, and external references — flag any found), encryption (PDF/A forbids encryption — flag if present), metadata (PDF/A requires XMP metadata with specific fields — check for dc:title, dc:creator, xmp:CreateDate, pdf:Producer), and file attachments (only allowed in PDF/A-3). Display results in a sidebar panel with a pass/fail status for each criterion, color-coded (green checkmark = pass, red X = fail, yellow warning = advisory). Show total compliance percentage and which PDF/A level the document currently meets (if any). Each failed criterion should include a brief explanation of what's wrong and whether auto-fix is available. **Auto-conversion**: Provide a "Convert to PDF/A" button that attempts to fix non-compliant issues automatically using pdf-lib. Supported auto-fixes: embed missing fonts (using standard Base14 font substitution where the original isn't available, with a warning about visual changes), strip JavaScript and actions, remove encryption metadata flags, add required XMP metadata fields (generate from existing info dictionary), flatten transparency (by rendering transparent layers to opaque bitmaps — use canvas rendering to composite and replace). For issues that can't be auto-fixed (e.g., complex font substitution), show a dialog explaining what needs manual attention. After conversion, re-run the validation to show the updated compliance status. **PDF/A level selection**: Let users choose the target PDF/A conformance level via a dropdown: PDF/A-1b (most restrictive, widest compatibility), PDF/A-2b (allows JPEG2000, transparency, layers), PDF/A-3b (allows file attachments). Show a tooltip explaining the differences between levels. Default to PDF/A-2b as a good balance of compatibility and features. **Output identification**: When converting, add the required PDF/A identification entry to the document catalog (GTS_PDFA1 for 1b, etc.) and embed the appropriate ICC color profile (sRGB) if not already present. Set the XMP metadata pdfaid:part and pdfaid:conformance values. **Batch validation**: Integrate with the batch processing system (TASK-030) so users can validate multiple PDFs at once and get a summary report showing which files are compliant and which need attention. Export the batch report as CSV or JSON. **Implementation approach**: Create a new module `js/pdfa.js` that handles both validation and conversion logic. For validation, parse the PDF's internal structure using pdf-lib's low-level API to inspect the catalog, metadata streams, font dictionaries, and page content streams.
