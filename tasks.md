### TASK-050: Headers, footers, and page numbering tool

**Status**: DONE
**Priority**: HIGH
**Assigned to**: developer
**Description**: Build a headers and footers tool that lets users add recurring text elements — page numbers, document title, date, author, and custom text — to every page (or a page range) of a PDF. This is one of the most commonly requested PDF editing features, essential for professional documents, reports, legal filings, and academic papers. **Header/footer zones**: Divide the top and bottom margins of each page into three zones each: left, center, and right — giving 6 configurable slots total (top-left, top-center, top-right, bottom-left, bottom-center, bottom-right). Each zone can contain a combination of static text and dynamic tokens. **Dynamic tokens**: Support placeholder tokens that auto-fill per page: `{page}` (current page number), `{pages}` (total page count), `{date}` (current date in configurable format: MM/DD/YYYY, DD/MM/YYYY, YYYY-MM-DD, or a custom date string), `{filename}` (the PDF filename without extension), and `{title}` (PDF document title from metadata, or filename if no title). Allow combining tokens with text, e.g., "Page {page} of {pages}" or "Confidential — {date}". **Page numbering styles**: Support multiple numbering formats: Arabic (1, 2, 3), Roman lowercase (i, ii, iii), Roman uppercase (I, II, III), alphabetic lowercase (a, b, c), and alphabetic uppercase (A, B, C). Allow a starting page number offset (e.g., start at page 3 to account for a cover page and table of contents). Provide a "Skip first page" toggle (common for documents with a cover page that shouldn't show a header/footer) and a configurable page range (e.g., apply to pages 5–50 only). Support different formatting for odd/even pages — e.g., page number on the right for odd pages and left for even pages (book-style alternating). **Font and style controls**: Font family (Helvetica, Times, Courier — standard PDF fonts), font size (6–24pt, default 10pt), font color (black default, with color picker), bold/italic toggles, and opacity (0.1–1.0). Provide separate styling per zone if the user enables "Advanced mode" — otherwise a single style applies to all zones. **Margin and positioning**: Configurable margin offsets — how far the header is from the top edge and the footer from the bottom edge (default 36pt / 0.5 inch). Horizontal margins from left/right edges (default 36pt). Show visual margin guides on the preview. **Separator line**: Optional thin horizontal line between the header and page content (and between page content and footer). Configurable line thickness (0.5–3pt), color, and style (solid, dashed). **Live preview**: Show a real-time preview of the header/footer on the current page as the user adjusts settings. Render the text overlays on the PDF viewer canvas, updating on every input change with 50ms debounce. Include a page navigator in the preview to verify how the headers/footers look on different pages (especially useful when odd/even alternating is enabled). **Implementation with pdf-lib**: Load the PDF via `PDFDocument.load()`. For each target page, resolve dynamic tokens by substituting `{page}` with the calculated page number (respecting the starting offset), `{pages}` with the total page count of the target range, `{date}` with the formatted current date, and `{filename}`/`{title}` from the document's info dictionary or the loaded filename. Embed the chosen font via `pdfDoc.embedFont(StandardFonts.Helvetica)` etc. For each zone, calculate the text position: top zones use `y = page.getHeight() - topMargin`, bottom zones use `y = bottomMargin - fontSize`. Left zones use `x = horizontalMargin`, center zones use `x = (page.getWidth() - textWidth) / 2`, right zones use `x = page.getWidth() - horizontalMargin - textWidth`. Compute text width using `font.widthOfTextAtSize(text, fontSize)`. Draw with `page.drawText()` and set color using `rgb()` from pdf-lib. For separator lines, use `page.drawLine()` spanning the page width between the horizontal margins. Handle pages with different sizes by recalculating positions per page. **Presets**: Provide one-click presets for common configurations: "Page numbers bottom center" (`Page {page} of {pages}` in bottom-center), "Title and date header" (title top-left, date top-right), "Legal footer" (page number bottom-right, confidential notice bottom-left), "Book style" (odd pages: title top-right, page bottom-right; even pages: title top-left, page bottom-left), and "Academic paper" (author top-left, title top-center, page bottom-center). Let users save custom presets to localStorage. **Remove existing headers/footers**: Provide a "Remove headers/footers" mode that detects and removes text in the header/footer margin zones. Scan each page's text content using pdf.js `page.getTextContent()`, identify text items positioned within the top or bottom margin area (within configurable threshold), display them for the user to confirm, and then use pdf-lib to redact those text items by drawing white rectangles over them before adding new headers/footers (since pdf-lib cannot directly remove content stream operators). **Batch mode**: Support applying the same header/footer configuration across multiple uploaded PDFs, processing them sequentially and downloading each as `[filename]-numbered.pdf`. **Undo**: Output as `[original-filename]-numbered.pdf` — preserve the original document in memory so the user can adjust settings and re-apply without reloading. **UI integration**: Add a "Header & Footer" tool as a new sub-panel under the "Pages" tab or as a standalone panel accessible from the toolbar. The panel shows the 6 zone input fields arranged visually to match their page positions (3 across the top, 3 across the bottom), with the style controls below and the preview to the right. Include a token insertion helper — small buttons next to each zone input that insert `{page}`, `{pages}`, `{date}`, etc. on click. **Keyboard accessibility**: Tab through all zone inputs and controls, ARIA labels on all interactive elements. **Edge cases**: Handle pages with existing headers/footers (warn and offer to remove first), pages with different sizes (recalculate positions per page), rotated pages (place headers/footers relative to the visual top/bottom considering rotation), very long header/footer text that exceeds page width (auto-truncate with ellipsis or auto-reduce font size to fit), page counts exceeding the alphabetic format (after Z, continue with AA, AB...), and right-to-left text. All processing client-side in the browser using pdf-lib.

(2) **From text analysis** — scan each page's text content using pdf.js `page.getTextContent()`, identify heading-like text by detecting significantly larger font sizes (>= 1.5× the dominant body font size), bold font names (containing "Bold"), or text matching common heading patterns (e.g., all-caps lines, "Chapter N", "Section N.N"). (3) **Manual entry** — let users type TOC entries manually, specifying title, page number, and indent level, useful for scanned documents or PDFs with no extractable structure. **TOC page layout**: Generate one or more TOC pages using pdf-lib. The layout includes: a title ("Table of Contents" or user-customizable), entries formatted as "Title ..... Page Number" with dot leaders (configurable: dots, dashes, underscores, or none) connecting the title to the right-aligned page number. Indent sub-entries by 20pt per level (configurable). Use a clean, professional typography: Helvetica for the title (16pt bold), Helvetica for entries (11pt, with bold for level-1 entries and regular for sub-entries), or let the user choose from Helvetica/Times/Courier. If entries exceed one page, automatically flow to additional TOC pages. **Clickable links**: Each TOC entry should be a clickable internal link that navigates to the referenced page. Use pdf-lib to create `/Link` annotations on each entry's text region with a `/GoTo` action pointing to the target page (`/A << /S /GoTo /D [pageRef /Fit] >>`). This makes the TOC functional in any PDF viewer that supports links. **Page number adjustment**: When TOC pages are inserted at the beginning, all existing page references shift. The generator must account for this: if the TOC occupies 2 pages, then what was page 1 becomes page 3. All TOC entry page numbers and link destinations must reference the adjusted page indices. If the document has existing headers/footers with page numbers (from TASK-050), warn the user that they may need to re-apply page numbering after TOC insertion. **Live preview**: Show a real-time preview of the generated TOC page(s) rendered on a canvas before inserting. Allow the user to toggle entries on/off via checkboxes in the detected headings list. **Customization panel**: Provide controls for: TOC title text (default "Table of Contents"), title font size (12–24pt), entry font size (9–14pt), line spacing multiplier, leader style (dots/dashes/none), margin size, whether to include page numbers, maximum depth level to include (1–4), and whether to bold level-1 entries. Include style presets: "Classic" (Times font, dot leaders, 1.2× spacing), "Modern" (Helvetica, no leaders, 1.5× spacing, left-aligned page numbers), and "Compact" (9pt, 1.0× spacing, dot leaders). **Update existing TOC**: If the document already has a TOC (detected by finding a page whose text starts with "Table of Contents" or "Contents" in the first few pages), offer to replace it rather than adding a duplicate. The panel shows the detected/entered TOC entries on the left and a preview on the right. Include "Detect from Bookmarks", "Detect from Text", and "Manual Entry" mode tabs. A "Generate & Insert" button creates the TOC pages and inserts them. **Output**: Insert the TOC page(s) at position 1 (before the first existing page) by default, or at a user-chosen position. Save the modified PDF; since pages are inserted, the output uses the original filename. Also offer "Export TOC only" to download just the TOC page(s) as a separate PDF. **Edge cases**: Handle documents with no detectable headings or bookmarks (show "No structure detected — use Manual Entry mode"), very long TOC entries that wrap to multiple lines (wrap text within the available width minus the page number column), page numbers exceeding 4 digits (adjust column width), documents with hundreds of headings (paginate the TOC across multiple pages, cap at 500 entries with a warning), and right-to-left text. All processing client-side in the browser using pdf-lib and pdf.js.

### TASK-053: PDF accessibility checker and compliance report generator

**Status**: DONE
**Priority**: MEDIUM
**Assigned to**: developer
**Description**: Build a PDF accessibility checker that analyzes documents against WCAG 2.1 and PDF/UA (ISO 14289) guidelines, then generates a detailed compliance report with actionable fix suggestions. This is an increasingly essential feature — government agencies, educational institutions, and many businesses are legally required to produce accessible PDFs (ADA Section 508, EU EN 301 549). **Structure analysis**: Use pdf.js to parse the document's structure tree (tagged PDF detection). Check whether the PDF contains a tag tree at all (`/MarkInfo << /Marked true >>` in the document catalog). If tagged, walk the structure tree to verify proper nesting: document must have a root `<Document>` tag, headings (`<H1>`–`<H6>`) must follow sequential order without skipping levels (e.g., H1 → H3 without H2 is a violation), paragraphs (`<P>`), lists (`<L>`, `<LI>`, `<Lbl>`, `<LBody>`), and tables (`<Table>`, `<TR>`, `<TH>`, `<TD>`) must be properly structured. Flag orphaned content not within any structural tag. **Image alt text**: Detect images (`<Figure>` tags or inline images without tags) and check for the presence of `/Alt` text entries. Flag images missing alt text with their page number and approximate position. For untagged PDFs, detect images via the page's operator list (looking for `paintImageXObject` operators) and flag all of them as inaccessible. **Reading order**: Analyze the logical reading order from the structure tree and compare it against the visual layout order derived from text item coordinates on each page. Flag pages where the tagged reading order significantly diverges from the expected left-to-right, top-to-bottom visual flow — compute this by comparing the sequence of text block Y-coordinates in the tag tree vs. the raw content stream order. **Language specification**: Check that the document has a `/Lang` entry in the catalog specifying the primary language (e.g., "en-US"). Flag if missing — screen readers need this to select the correct pronunciation engine. Also detect spans marked with a different `/Lang` attribute for multilingual documents. **Color contrast**: For text elements, extract the font color and estimate the background color (from the page's background fill or the nearest filled rectangle behind the text). Compute the WCAG contrast ratio (relative luminance formula). Flag text below 4.5:1 ratio for normal text or 3:1 for large text (>= 18pt or >= 14pt bold). Display failing elements with their actual contrast ratio and the minimum required. **Form field labels**: If the PDF contains form fields (AcroForm), check each field for an associated `/TU` (tooltip/accessible name) entry. Flag fields without accessible names — screen readers cannot describe unlabeled form fields to users. **Bookmarks**: Check whether the document has an outline/bookmark tree. For documents longer than 5 pages, flag missing bookmarks as a navigation accessibility issue since long documents need bookmarks for efficient screen reader navigation. **Title**: Verify the document's `/Title` metadata is set and that the viewer is configured to display the title rather than the filename (`/ViewerPreferences << /DisplayDocTitle true >>`). **Report generation**: Display results in a dedicated "Accessibility" panel as a categorized checklist with pass/fail/warning indicators: (1) Document Structure — tagged PDF, tag nesting, reading order. (2) Text Alternatives — image alt text, figure descriptions. (3) Color & Contrast — contrast ratios, color-only information. (4) Navigation — bookmarks, heading hierarchy, page labels. (5) Forms — field labels, tab order. (6) Metadata — language, title, display title. Each failed check shows the issue description, affected page(s), severity (critical/major/minor), the relevant WCAG success criterion (e.g., "1.1.1 Non-text Content"), and a suggested fix. Show an overall compliance score as a percentage with a color-coded badge (red < 50%, yellow 50-80%, green > 80%). **Export report**: Let users download the accessibility report as a PDF (generated via pdf-lib with the check results formatted as a structured document) or as a JSON file (machine-readable, useful for automated compliance workflows). The PDF report should include the document filename, check date, overall score, and the full categorized results. **Page-level drill-down**: Clicking a failed check navigates to the affected page in the viewer and highlights the problematic element (e.g., an image missing alt text gets a red dashed border overlay). **Quick fixes**: Where possible, offer one-click fixes within the checker panel: add missing `/Lang` to the document catalog, set `/DisplayDocTitle` to true, add `/TU` tooltips to form fields (user types the label), add `/Alt` text to figure tags (user types the description). Apply fixes using pdf-lib and allow the user to re-run the check to verify improvements. Save the fixed PDF as "-accessible.pdf". **UI integration**: Add an "Accessibility" button in the main toolbar or as a new top-level tab. The panel shows a summary dashboard at the top (overall score, pass/fail/warning counts) and a scrollable categorized checklist below. Include a "Run Check" button that triggers the full analysis with a progress indicator. For large documents, analyze pages in batches to avoid UI freezing (use `requestIdleCallback` or chunked `setTimeout`). **Edge cases**: Handle completely untagged PDFs gracefully (report "Document is not tagged — most accessibility checks cannot be performed" and suggest the user add tags using a dedicated PDF authoring tool), scanned image-only PDFs (detect via lack of extractable text, suggest running OCR first via TASK-008), encrypted PDFs (warn that some checks may be limited), and very large documents (> 500 pages) by offering a "Quick scan" mode that samples every Nth page. All processing client-side in the browser using pdf.js for structure parsing and pdf-lib for applying fixes.

### TASK-054: Export PDF to HTML with responsive layout and embedded assets

**Status**: DONE
**Priority**: HIGH
**Assigned to**: developer2
**Description**: Build a PDF-to-HTML export feature that converts PDF pages into a self-contained, responsive HTML document preserving text content, layout structure, images, and basic formatting — completing the conversion toolkit alongside the existing PDF-to-Images (TASK-014), PDF-to-Word (TASK-038), and Table-to-CSV (TASK-037) exporters. This is essential for publishing PDF content on the web, making documents searchable by search engines, and creating accessible web versions of print-formatted documents. **Text extraction and structure mapping**: Use pdf.js `page.getTextContent()` to extract all text items with their positions (x, y, width, height), font sizes, font names, and colors. Map extracted content to semantic HTML elements: detect headings by identifying text with significantly larger font sizes (>= 1.5× the dominant body font size) or bold font names — map these to `<h1>`–`<h6>` based on relative size ranking. Group text items into paragraphs by analyzing vertical spacing (items with Y gaps < 1.5× font size belong to the same paragraph → `<p>` tags). Detect lists by checking for leading bullet characters (•, -, ●) or numbered patterns (1., 2., a.) → `<ul>`/`<ol>` with `<li>` children. Detect tables by identifying grid-aligned text clusters (reuse the column detection logic from TASK-037's table extraction) → `<table>` with `<thead>`/`<tbody>`/`<tr>`/`<td>`. **Image extraction**: Extract embedded images from each page using pdf.js's operator list (identify `paintImageXObject` and `paintInlineImageXObject` operators), render each image to an offscreen canvas, and export as base64 PNG data URIs for self-contained output, or as separate PNG/JPG files for the multi-file export mode. Position images relative to surrounding text using CSS. For pages where text extraction yields poor results (scanned/image-heavy pages), fall back to rendering the entire page as a high-resolution image and embedding it in the HTML with an `<img>` tag. **Layout modes**: Provide two export modes the user can choose between: (1) **Reflowable HTML** (default) — text is extracted into semantic HTML elements that reflow naturally with the browser window width, suitable for reading on any device. CSS sets a max-width of 800px centered on the page, with responsive font sizing. This produces the cleanest, most accessible output but may not perfectly match the PDF's visual layout. (2) **Fixed-layout HTML** — each page is rendered as a positioned `<div>` with absolute CSS positioning for every text element, closely matching the PDF's exact visual layout at the expense of responsiveness. Useful when precise layout fidelity matters (forms, certificates, technical drawings). Each page div has the PDF page's aspect ratio and a configurable width (default 800px). **Styling**: Generate clean, well-structured CSS: use CSS custom properties (variables) for colors and fonts to make the output easy to customize. Map PDF font names to web-safe font stacks (e.g., Helvetica → `'Helvetica Neue', Arial, sans-serif`, Times → `'Times New Roman', Georgia, serif`, Courier → `'Courier New', monospace`). Preserve text colors via inline `style` or CSS classes. Apply bold/italic based on font name analysis. Add page breaks between PDF pages using `page-break-after: always` for print compatibility. Include a responsive viewport meta tag and basic print stylesheet. **Navigation**: Generate a navigation sidebar or top bar listing each page (and detected headings) as clickable anchor links for quick navigation within the HTML document. Include a "Back to top" floating button. If the PDF has bookmarks, convert them to an HTML table of contents with anchor links. **Output formats**: (1) **Single HTML file** — everything self-contained in one `.html` file with inline CSS, inline base64 images, and inline JavaScript (for the navigation sidebar toggle). Ideal for sharing a single file. Warn if the file exceeds 10MB (large PDFs with many images).
